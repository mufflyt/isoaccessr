Executive Summary

IsoAccessR is a lightweight R package designed to standardize and simplify geographic accessibility analysis. The project addresses a common and recurring challenge across domains such as healthcare, public health, transportation, and policy: estimating population access to services using travel-time service areas.

Currently, analysts must manually implement area-weighted overlays to estimate population coverage from service-area polygons. This process is technically complex, error-prone, and frequently reimplemented across projects, leading to inconsistent methods and limited reproducibility. While the R ecosystem provides strong foundational spatial tools, there is no focused, reusable package that standardizes this workflow.

IsoAccessR will fill this gap by providing a minimal, well-defined interface for accessibility analysis. The package will enable users to combine service-area geometries with population data to produce analysis-ready summaries with minimal custom spatial code. The design emphasizes simplicity, transparency, and interoperability with the modern R spatial ecosystem, particularly sf and tidyverse workflows.

The proposed minimum viable product (MVP) focuses on a single, high-value workflow and will include two core functions: iso_estimate_access() for calculating population access using area-weighted overlays, and iso_summarize_access() for aggregating results across geographies. The package will also include an example dataset, a reproducible end-to-end vignette, and automated testing with continuous integration.

The project builds on prior internal work that has successfully implemented accessibility estimation across multiple geographies and time periods, reducing technical risk and ensuring feasibility. Development will occur over a four-month timeline, including design, implementation, documentation, and refinement phases.

Expected outcomes include a publicly available open-source R package, a pkgdown documentation site, and reproducible workflows that can be used and extended by analysts across multiple disciplines. By reducing the need for bespoke spatial pipelines, IsoAccessR will improve reproducibility, consistency, and efficiency in accessibility analysis.

The total requested budget is $8,000 USD, allocated across development, documentation, testing, and community feedback. This budget aligns with prior R Consortium Infrastructure Steering Committee grants for focused package development.

IsoAccessR represents a small but impactful addition to the R ecosystem, providing a standardized, reusable tool for a widely needed analytical task.

The Problem

Geographic accessibility analysis is a common task across healthcare, public health, transportation, and policy. Analysts frequently need to estimate how many people can reach a service within a defined travel time, using service-area polygons (e.g., isochrones) and population data. Despite its importance, there is no lightweight, standardized R package that supports this workflow from spatial inputs to analysis-ready summaries.

This problem affects a broad group of R users, including health services researchers, epidemiologists, policy analysts, and data scientists. Many of these users are proficient in R but do not have deep expertise in spatial data engineering. As a result, they must assemble custom workflows using lower-level spatial tools to perform overlay operations, area-weighted interpolation, and aggregation. These steps are technically complex and often repeated across projects.

A central challenge is estimating population coverage from service areas. This typically requires implementing area-weighted overlays between service polygons and population geometries (e.g., census tracts). In practice, this process is error-prone, sensitive to methodological assumptions, and rarely implemented in a consistent or reusable way. Small differences in implementation—such as how overlaps are calculated or aggregated—can lead to materially different results, reducing comparability across studies.

The consequences are fragmented workflows, limited reproducibility, and high setup costs for new analyses. Even within the R ecosystem, users must rely on combinations of powerful but low-level packages such as sf for spatial operations and dplyr for data manipulation. While these tools are essential, they do not provide a cohesive, opinionated interface for accessibility analysis. Analysts therefore reimplement similar logic repeatedly, increasing the likelihood of errors and reducing efficiency.

Solving this problem would enable a standardized, reproducible approach to accessibility analysis in R. A focused package that encapsulates core operations—such as area-weighted estimation and aggregation—would reduce the need for bespoke code, improve consistency across analyses, and lower the barrier to entry for non-specialist users. It would also facilitate more transparent and auditable workflows, which are particularly important in policy-relevant applications.

Existing work in the R ecosystem provides strong foundations but does not directly address this gap.[@pebesma2018] Packages such as sf provide robust spatial data structures and operations, and the tidyverse enables flexible data manipulation workflows [@wickham2016]. Other packages support related tasks such as data access or mapping, but none offer a minimal, dedicated interface for estimating population access from service areas. As a result, accessibility analysis remains a repeated, custom-built solution rather than a standardized workflow.

IsoAccessR is designed to address this gap by providing a lightweight, reusable layer for accessibility analysis that builds on existing tools while standardizing a widely needed analytical task.

The proposal

IsoAccessR will address the lack of a standardized accessibility workflow by delivering a lightweight R package focused on estimating population access from service-area polygons using area-weighted overlays. The project will translate a commonly reimplemented but error-prone spatial workflow into a simple, reusable interface that integrates cleanly with the existing R spatial ecosystem.

The proposal focuses on a minimal, high-impact solution rather than a full platform. By standardizing a single core workflow, IsoAccessR will reduce duplicated effort, improve reproducibility, and lower the barrier for analysts who need to perform accessibility analysis without building custom spatial pipelines.

Development will proceed over a four-month period, with clear milestones for design, implementation, documentation, and refinement. The result will be a publicly available R package with a small, well-defined API, a reproducible vignette, and automated tests.

Overview

IsoAccessR will provide a compact interface for a widely needed task: estimating how many people can access a service within a defined travel time. It will do so by combining service-area polygons with population geometries using area-weighted overlays and returning tidy, analysis-ready outputs.

This proposal addresses the problem of fragmented and inconsistent accessibility workflows by introducing a focused abstraction layer above existing spatial tools such as sf. Rather than replacing these tools, IsoAccessR will make them easier to use for a specific analytical purpose.

Concrete actions include: - designing a minimal API for accessibility estimation

Detail

IsoAccessR focuses on a single, high-value workflow: estimating population access from service areas. The package will encapsulate the key steps—overlay, weighting, and aggregation—into a minimal set of functions, reducing the need for custom spatial code.

Minimum Viable Product

The MVP will include:

  • iso_estimate_access()
    Computes population access using area-weighted overlays between service-area polygons and population geometries

  • iso_summarize_access()
    Aggregates access estimates across user-defined geographies

  • one example dataset

  • one reproducible end-to-end vignette

  • automated tests and continuous integration

This MVP is intentionally scoped to deliver immediate value while remaining feasible within the project timeline.

Architecture

The package will follow a simple and transparent structure:

  • Inputs:
    • service-area polygons (sf objects)
    • population geometries with counts
  • Processing:
    • spatial intersection
    • area-weighted calculation of population coverage
    • aggregation across geographies
  • Outputs:
    • tidy tabular summaries compatible with tidyverse workflows

This design avoids unnecessary abstraction while ensuring interoperability with existing spatial tools.

Assumptions

The project assumes:

  • users already have access to service-area (travel-time) polygons
  • spatial geometries are valid and share a common coordinate reference system
  • population data is available at a defined geographic level

If these assumptions are violated, results may be invalid; however, these constraints are standard in spatial analysis workflows and consistent with existing tools.

External dependencies

IsoAccessR will build on established R packages:

  • sf for spatial data operations
  • dplyr for data manipulation
  • testthat for testing
  • pkgdown for documentation

These dependencies are widely used, stable, and well-supported within the R ecosystem, minimizing risk and ensuring long-term compatibility.

Success

The project will be considered complete when the following deliverables are met:

  1. Public R package available
    • IsoAccessR is hosted in a public GitHub repository
    • Package is installable via devtools::install_github()
  2. Core functionality implemented
    • iso_estimate_access() correctly performs area-weighted accessibility estimation
    • iso_summarize_access() aggregates results across geographies
  3. Reproducible workflow provided
    • A complete vignette demonstrates end-to-end usage with example data

  4. Testing and reliability
    • Unit tests are implemented using testthat
    • Continuous integration (CI) passes consistently
  5. Documentation and usability
    • Function documentation is complete and clear

Future work
